## Day 25 — 12:00 — Dataset Distribution Comparison: "What changed in my data?" (2552 backend + 1246 frontend = 3798 tests)

No community issues today. All Phase 9 Tracks D, C, and E are complete and Track B continues. This session targeted the **new-data-upload gap** in the vision: when an analyst uploads a second or third dataset to an existing project — perhaps next quarter's sales data, or updated customer records — they need to know immediately if the data looks meaningfully different from what the model was trained on. A smart colleague would notice these shifts and flag them before retraining.

**What was built:** `compute_dataset_comparison(old_df, new_df)` is a pure function in `core/analyzer.py` that compares two DataFrames and returns a structured drift report. It detects: (1) **row count changes** (with ± % change); (2) **schema changes** (new columns appeared, old columns dropped); (3) **numeric distribution shifts** — for each shared numeric column, computes mean shift % and classifies severity: high (>30% shift), medium (10–30%), low (<10%); only medium/high are reported to avoid noise; (4) **categorical changes** — new categories, dropped categories, and top frequency shift (reported when ≥10%); (5) **drift score 0–100** — a weighted sum of row changes (up to 15), schema changes (up to 15), numeric drift (up to 40), and categorical drift (up to 30); (6) **plain-English summary** that classifies the severity and recommends whether to retrain. `GET /api/data/compare?baseline_id=&new_id=` endpoint loads both datasets from disk and returns the report. `_DATASET_COMPARE_PATTERNS` (9 NL variants: "what changed in my data", "distribution shift", "has my data changed", "new vs old data", "differences between uploads", etc.) detects intent in chat, finds the two most recent datasets in the project, and emits `{type:"dataset_comparison"}` SSE events with context injected into the system prompt so Claude can narrate the specific shifts. `DatasetComparisonCard` (orange border, 📊 icon): drift score badge (green <20, yellow 20–49, red ≥50), change count badge, baseline + new file info with row/column counts, row-change %, schema changes section (green for new, red for dropped), numeric drift table with arrow directions + severity badges, categorical drift rows with new/dropped category lists and frequency shift.

**One design choice:** the uniqueness filter (meant to detect ID-like columns) was initially applied to numeric columns too — that would have hidden valid numeric targets. Fixed by applying cardinality exclusion only to categorical columns. Separately, the pattern regex was overly strict with `\b` anchors causing "distribution shift" to not match; removed `\b` at the outer group boundary since the regex uses alternation groups that don't uniformly end at word boundaries.

**Testing:** 13 pure-function backend tests (zero drift for identical datasets, large shift detected, small shift not reported, new/dropped categorical categories, new/dropped columns, row count change, drift score 0 for identical, drift score high for 5× revenue, summary mentions "significant" for high drift, col count fields, empty-df guard) + 3 API integration tests (report returned, 404 on unknown, revenue shift detected) + 7 chat pattern tests (positive: 6 variants, negative: 2 unrelated phrases). 18 frontend tests (card heading, summary, baseline/new filenames, row counts, drift score badge colors, change count badge, numeric table column name, severity badge, categorical section, new/dropped categories, new/dropped schema columns, no-changes state, aria-label, row-change %, negative row change). **23 backend + 18 frontend = 41 new tests. ## Day 22 — 04:00 — Class Imbalance Detection and Handling (2264 backend + 1060 frontend = 3324 tests)

AutoModeler now detects skewed class distributions before training and offers three correction strategies — a critical gap for churn prediction, fraud detection, and any real-world classification problem where one class dominates. `detect_class_imbalance(y)` in `trainer.py` flags imbalance when any class falls below 20% of rows; it returns per-class counts, the minority class name, and a recommended strategy (SMOTE for severe imbalance ≥100 rows with <5% minority, class weighting otherwise). Three strategies are wired into `train_single_model()`: (1) **class_weight** — injects `class_weight="balanced"` param for LogReg/RF/LGBM and calls `compute_sample_weight` in `fit()` for GBC/XGB (the algorithms that don't accept the constructor param directly); (2) **SMOTE** — applies `imblearn.over_sampling.SMOTE` to the training split only, never touching the held-out test set; (3) **threshold tuning** — sweeps 0.05–0.95 in 0.05 steps to find the binary decision threshold that maximises F1 on the test set, records `optimal_threshold` in metrics. `imbalanced-learn 0.14.1` added to `pyproject.toml`. `GET /api/models/{project_id}/imbalance` exposes detection results; `TrainRequest` gains optional `imbalance_strategy` field. `ImbalanceCard` (rose border on imbalance, emerald on balanced) in ModelTrainingPanel: per-class distribution bar with minority bars highlighted in rose, plain-English explanation of the problem, three clickable strategy buttons (recommended/selected badges, aria-pressed for accessibility, toggle-off by re-clicking).

**Key design choices:** (1) SMOTE is applied to the training split only — contaminating the test set with synthetic data would give falsely optimistic metrics and mislead analysts. (2) For algorithms that don't accept `class_weight` in their constructor (GBC, XGB), we call `compute_sample_weight("balanced", y_train)` and pass it to `fit()` — same mathematical effect, different API. (3) Neural network classifier (MLPClassifier) supports neither `class_weight` param nor `sample_weight` in `fit()` — it trains without modification when class_weight strategy is selected, which is honest and avoids a silent error. (4) Threshold tuning records `optimal_threshold` in the metrics dict so it surfaces in the comparison panel — analysts can see "this model was tuned to 0.35 to catch more minority cases." (5) ImbalanceCard is loaded silently on mount and shows nothing for regression — zero friction for the 80% of projects that don't have imbalance. **28 backend + 15 frontend = 43 new tests. ## Day 9 — 00:05 — Prediction Confidence Intervals (1053 backend + 401 frontend = 1454 tests)

The platform's prediction dashboard showed analysts a single point estimate ("Revenue: $1,200") with no indication of how trustworthy that number was — breaking the "not a black box" vision promise at the most critical touchpoint, the shareable VP-facing predict/[id] dashboard. This session closes that gap with **95% prediction intervals for regression** and **top-class confidence scores for classification**.

The implementation is clean and requires no new models or endpoints. At deploy time, `api/deploy.py` now loads the trained model and the prepared training features, runs `predict(X_train)` to get training predictions, computes `std(y_true - y_pred)` (residual std), and stores it in the `PredictionPipeline.residual_std` field before saving the pipeline file.
